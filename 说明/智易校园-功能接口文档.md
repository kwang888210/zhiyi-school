## 系统概述

### 🎯 项目定位

“智易校园”是一个**面向高校学生的二手/需求交易平台**，核心差异化竞争力在于：

| 维度 | 亮点 |
|------|------|
| 🤖 **AI 智能审核** | 接入大模型 API，自动识别违禁内容 + 智能提取商品标签 |
| 🔒 **担保交易** | 平台托管资金，收货确认后打款，保障买卖双方权益 |
| 🌱 **成长体系** | 经验值 + 等级 + 信誉分，激励用户诚信交易 |
| 💬 **内置聊天** | 无需跳转第三方 IM，买卖双方站内沟通 |
| 🛡️ **风控惩罚** | 警告 → 限时封禁 → 永久封号，多级违规惩戒 |

## 模块一：用户认证与成长体系

> **负责人：A** | **涉及表：`sys_user`、`violation_log`** | **难度：⭐⭐⭐**

### 1.1 账户注册

**功能描述**：新用户通过学号完成注册，成为平台合法用户。

**详细流程**：

1. 用户访问注册页面，填写以下信息：
   - **学号**（登录凭证，唯一，必填）
   - **密码**（不少于 6 位，必填）
   - **确认密码**（二次校验）
   - **昵称**（展示用，默认生成为 `同学_学号后4位`，可修改）
   - **密保问题**（从系统预设问题中选择一条，如"你的小学名称是？"）
   - **密保答案**（用户自定义，用于密码找回）
   - **手机号**（可选，用于通知）
2. 前端校验密码长度 ≥ 6 位，且两次输入一致。
3. 后端校验学号唯一性，若已注册则提示"该学号已注册，请直接登录或找回密码"。
4. 密码使用 **BCrypt** 加密后存入数据库（不得明文存储）。
5. 注册成功，自动分配初始数据：
   - `level = 1`
   - `exp = 0`
   - `wallet_balance = 0.00`
   - `role = 'USER'`
   - `status = 'ACTIVE'`

**验收标准**：
- [ ] 学号唯一性校验通过
- [ ] 密码加密存储
- [ ] 注册后可直接登录

---

### 1.2 账户登录

**功能描述**：已注册用户使用学号 + 密码登录系统。

**详细流程**：

1. 用户输入学号 + 密码。
2. 后端查询 `sys_user` 表，验证密码。
3. 登录前检查账户状态：
   - `status = 'BANNED_TEMP'`：检查 `ban_until_time` 是否已过期。若未过期，拒绝登录并提示"您的账户已被封禁至 XXXX年XX月XX日"。
   - `status = 'BANNED_PERM'`：拒绝登录，提示"该账户已被永久封禁"。
   - `status = 'ACTIVE'`：正常登录。
4. 登录成功，生成 **JWT Token**（有效期 24 小时），返回前端存储于 localStorage。
5. 后续请求在 Header 携带 Token，后端通过拦截器校验。

**验收标准**：
- [ ] 封禁用户无法登录
- [ ] 临时封禁到期后自动恢复登录权限
- [ ] JWT 过期后需重新登录

---

### 1.3 密码找回

**功能描述**：用户忘记密码时，通过预设的密保问题重置密码。

**详细流程**：

1. 用户在登录页点击"忘记密码"。
2. 输入学号，系统查询该用户已设置的**密保问题**并回显。
3. 用户输入密保答案，系统比对（忽略首尾空格、不区分大小写）。
4. 验证通过后，跳转至密码重置页面，输入新密码并确认。
5. 重置成功后，清除该用户的所有活跃 Token，强制重新登录。

**设计考量**：
- 密保机制避免了接入短信/邮件服务的复杂度和成本。
- 密保答案也需经过 BCrypt 加密存储，不存明文。

**验收标准**：
- [ ] 密保答案验证正确才能重置密码
- [ ] 答案比对忽略大小写和首尾空格
- [ ] 重置后旧 Token 失效

### 1.4 用户角色划分

**功能描述**：系统内置两种角色，不同角色看到不同的前端界面和功能入口。

| 角色 | 标识 | 权限范围 |
|------|------|----------|
| **普通用户** | `role = 'USER'` | 买卖商品、聊天、充值、查看个人主页 |
| **系统管理员** | `role = 'ADMIN'` | 审核商品、封禁用户、查看数据大盘、处理客诉、强制下架 |

**实现要点**：
- 管理员账户在数据库初始化时通过 SQL 脚本直接插入，不由前端注册产生。
- 前端根据 `role` 字段渲染不同的导航菜单（普通用户看不到"管理控制台"入口）。
- 后端每个需要管理员权限的接口加 `@RoleRequired("ADMIN")` 注解校验。

**验收标准**：
- [ ] 管理员可访问管理后台，普通用户无法访问
- [ ] 普通用户无法通过直接调用 API 绕过权限

### 1.5 成长体系：经验值与等级

**功能描述**：用户通过完成交易获取经验值，累积经验值提升等级。等级是用户诚信度的直观体现。

**经验值获取规则**：

| 行为 | 经验值增量 | 说明 |
|------|-----------|------|
| 作为买家完成一笔订单 | +50 EXP | 确认收货后发放 |
| 作为卖家完成一笔订单 | +50 EXP | 买家确认收货后发放 |
| 商品被管理员强制下架 | -30 EXP | 违规惩罚 |

**等级升级规则**：

| 等级 | 所需累计经验 | 称号 |
|------|-------------|------|
| Lv.1 | 0 | 初来乍到 |
| Lv.2 | 100 | 小有信誉 |
| Lv.3 | 300 | 值得信赖 |
| Lv.4 | 600 | 交易达人 |
| Lv.5 | 1000 | 校园传奇 |

**展示位置**：
- 商品详情页（卖家昵称旁显示等级徽章）
- 聊天窗口（对方头像旁显示等级）
- 个人主页（显示等级进度条）

**实现要点**：
- `exp` 字段记录累计经验值，`level` 字段记录当前等级。
- 每次经验值变动后，检查是否触发升级（`exp >= 下一级所需阈值`），若触发则更新 `level` 并推送通知。
- 经验值只增不减（除违规惩罚外），降级不做回退处理。

**验收标准**：
- [ ] 完成订单后经验值自动增加
- [ ] 达到阈值自动升级
- [ ] 等级在商品详情和聊天中正确展示

### 1.6 多级违规与封禁惩罚机制

**功能描述**：管理员可对违规用户实施分级惩罚，基于数据库时间戳自动控制权限。

**惩罚等级**：

| 等级 | 操作 | 数据库变化 | 效果 |
|------|------|-----------|------|
| **警告** | 发送系统通知 | `violation_log` 插入记录，`violation_type = 'WARNING'` | 用户收到提醒，不影响使用 |
| **限时封禁** | 封禁 N 天 | `status = 'BANNED_TEMP'`，`ban_until_time = 当前时间 + N天` | 封禁期间无法登录 |
| **永久封号** | 冻结账户 | `status = 'BANNED_PERM'` | 永远无法登录 |

**实现要点**：
- 封禁操作记录到 `violation_log` 表：记录违规用户 ID、操作管理员 ID、违规类型、封禁原因、处罚时长。
- 登录时自动校验 `ban_until_time` 与当前时间：若已过期，自动恢复 `status = 'ACTIVE'` 并允许登录。
- 管理员可手动"提前解封"，将 `status` 改回 `ACTIVE` 并清空 `ban_until_time`。

**验收标准**：
- [ ] 限时封禁到期后用户可自动恢复登录
- [ ] 永久封禁用户永远无法登录
- [ ] 所有处罚记录可追溯

## 模块二：AI 赋能的内容发布中枢

> **负责人：B** | **涉及表：`item`、`category`、`violation_report`** | **难度：⭐⭐⭐⭐**

### 2.1 商品分类体系

**功能描述**：系统预设商品大类，用户发布时从固定大类中选择，AI 自动生成小标签实现精细化分类。

**预设大类**（`category` 表初始化数据）：

| 大类名称 | 图标 | 典型子类（AI 动态标签示例） |
|----------|------|---------------------------|
| 📱 数码电子 | 💻 | 手机、平板、笔记本、耳机、充电宝、键盘 |
| 📚 教材书籍 | 📖 | 高数、英语四级、考研政治、计算机基础 |
| 👕 服饰鞋包 | 👟 | T恤、外套、运动鞋、背包、帽子 |
| 🏠 生活日用 | 🛏️ | 台灯、收纳箱、风扇、衣架、镜子 |
| 🎮 运动娱乐 | ⚽ | 篮球、瑜伽垫、滑板、吉他、Switch |
| 🍜 零食饮品 | 🥤 | 泡面、咖啡、辣条、坚果 |
| 📝 学习用品 | ✏️ | 笔记本、笔、计算器、尺子 |
| 🔧 其他 | 📦 | 以上均不适用的物品 |

**核心设计思路**：
- **大类固定**：用户只需选择 1 个大类，降低发布门槛。
- **小类动态**：AI 从描述中智能提取标签（见 2.4），存入 `item.ai_tags` 字段（JSON 数组），实现动态分类。
- 搜索时，大类使用精确匹配（`WHERE category_id = ?`），小类使用模糊搜索（`LIKE '%tagname%'`）。

**验收标准**：
- [ ] 发布商品时大类必选
- [ ] 管理员可增删改大类
- [ ] 搜索可按大类筛选

### 2.2 双向发布流

**功能描述**：用户可发布两种类型的帖子——"我要出闲置"（卖方）和"我要求购"（买方）。

**发布表单字段**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 发布类型 | 单选 | ✅ | `SELL`（出售）/ `BUY`（求购） |
| 商品标题 | 文本 | ✅ | ≤ 50 字 |
| 商品描述 | 文本域 | ✅ | ≤ 500 字，富文本 |
| 所属大类 | 下拉选择 | ✅ | 从预设大类中选择 |
| 商品图片 | 文件上传 | ✅ | 至少 1 张，最多 9 张 |
| 价格 | 数字 | ✅ | 精确到小数点后 2 位，≥ 0.01 |
| 交易地点 | 文本 | ✅ | 如"图书馆门口"、"食堂三楼" |

**商品生命周期**：

```
┌──────────┐    买家点击购买    ┌──────────┐   买家确认收货   ┌──────────┐
│  在售中   │ ───────────────→ │  交易中   │ ─────────────→ │  已售出   │
│ ON_SALE  │                  │ PENDING  │                │  SOLD    │
└──────────┘                  └──────────┘                └──────────┘
     │                             │
     │ 卖家/管理员下架             │ 买家取消订单
     ▼                             ▼
┌──────────┐                  ┌──────────┐
│  已下架   │                  │  在售中   │
│ OFF_SHELF│                  │ ON_SALE  │
└──────────┘                  └──────────┘
```

**验收标准**：
- [ ] 出售和求购两种类型均可正常发布
- [ ] 状态流转逻辑正确
- [ ] 已售出/已下架商品不再在首页展示

### 2.3 商品图片上传

**功能描述**：用户上传商品照片，系统存储至服务器并在商品详情页展示。

**实现要点**：

- 支持图片预览：发布前可查看已上传图片的缩略图。
- 首张图片自动设为主图（封面），在商品列表卡片中展示。

**验收标准**：
- [ ] 支持 jpg/png/webp 格式
- [ ] 单张不超过 5MB
- [ ] 图片正常加载展示

### 2.4 AI 智能机审与动态打标 🔥（核心亮点）

**功能描述**：用户提交发布申请时，系统自动调用大模型 API（如通义千问 / 文心一言 / GPT）对商品内容进行分析，实现**合规审核** + **智能打标**。

**完整流程**：

```
用户提交发布
      │
      ▼
┌─────────────────────────────────────────────┐
│  后端组装 Prompt：                           │
│  "你是一个校园交易审核助手。请分析以下商品：   │
│   标题：{title}                              │
│   描述：{description}                        │
│  请返回 JSON 格式：                          │
│  {                                          │
│    'is_violation': true/false,              │
│    'violation_reason': '...',               │
│    'tags': ['标签1', '标签2', ...]           │
│  }"                                         │
└─────────────────────────────────────────────┘
      │
      ▼
┌─────────────────────────────────────────────┐
│  调用大模型 API，解析返回结果                  │
└─────────────────────────────────────────────┘
      │
      ├── is_violation = true ──────────────────┐
      │                                         │
      │   ┌────────────────────────────────┐     │
      │   │ 1. 拦截发布，返回错误提示给用户  │     │
      │   │ 2. 写入 violation_report 表     │     │
      │   │ 3. 记录违规原因和原始文本        │     │
      │   └────────────────────────────────┘     │
      │                                         │
      └── is_violation = false ─────────────────┐
                                                │
          ┌────────────────────────────────┐     │
          │ 1. 提取 tags 数组              │     │
          │ 2. tags 存入 item.ai_tags (JSON)│     │
          │ 3. 商品正常发布，状态 = ON_SALE  │     │
          │ 4. 通知管理员"有新商品发布"      │     │
          └────────────────────────────────┘     │
```

**违规内容判定范围**（写入 Prompt）：

| 违规类型 | 说明 | 示例 |
|----------|------|------|
| 违禁品 | 危险品、管制物品 | 刀具、化学品 |
| 代考代写 | 学术不端服务 | "代写论文"、"代考" |
| 人身攻击 | 污言秽语、谩骂 | 含侮辱性词汇 |
| 虚假信息 | 明显欺诈 | "100元出iPhone 15 Pro" |
| 无关内容 | 与校园交易无关 | 广告、引流 |

**AI 动态标签示例**：

| 用户输入 | AI 生成标签 |
|----------|------------|
| "99新苹果平板 iPad Air5，考研结束出" | `["iPad", "苹果", "99新", "平板"]` |
| "高数第六版，只用了一学期，有笔记" | `["高数", "教材", "二手书", "有笔记"]` |
| "全新未拆封小米充电宝20000mAh" | `["充电宝", "小米", "全新", "20000mAh"]` |

**异常处理**：
- 若 AI 接口调用超时（设置 5 秒超时）或返回格式异常，**默认放行**但标记 `ai_reviewed = false`，同时写入 `violation_report` 标记为"待人工复核"，不影响用户体验。
- 管理员可在后台查看"待人工复核"列表并手动处理。

**验收标准**：
- [ ] 明显违禁内容被 AI 拦截
- [ ] 合规内容自动打上标签
- [ ] AI 超时时商品不卡死，降级放行
- [ ] 管理员可查看被拦截和待复核记录

## 模块三：交易大厅与社交互动

### 3.1 首页商品大厅

**功能描述**：用户登录后看到的首页，展示所有在售商品的信息流。

**布局设计**：

```
┌──────────────────────────────────────────────┐
```

### 3.5 近期爆款榜单

```sql
SELECT i.*, COUNT(f.id) AS fav_count
FROM item i
LEFT JOIN item_favorite f ON i.id = f.item_id

GROUP BY i.id
ORDER BY fav_count DESC
```

### 4.2 担保交易流程 🔥（核心技术难点）

**功能描述**：平台作为担保方，买家付款时资金冻结在平台，买家确认收货后资金才解冻打给卖家。这是向老师展示 **数据库事务（ACID）** 能力的绝佳模块。

**完整交易流程**：

```
步骤1：买家下单
┌──────────────────────────────────────────────────────┐
│ 买家在商品详情页点击"立即购买"                          │
│ → 系统创建订单，状态 = WAITING_MEET（待线下见面）       │
│ → 买家钱包余额 -= 商品价格（资金进入"平台担保"状态）     │
│ → 写入 wallet_log：PAYMENT，金额 = -商品价格            │
│ → 商品状态：ON_SALE → PENDING（交易中，他人不可再购买）  │
│                                                      │
│ 🔒 以上所有数据库操作必须在同一个 Transaction 中完成！    │
└──────────────────────────────────────────────────────┘
      │
      ▼
步骤2：线下见面交易
┌──────────────────────────────────────────────────────┐
│ 买卖双方通过聊天约定地点，当面验货                       │
│ → 无需系统操作                                        │
└──────────────────────────────────────────────────────┘
      │
      ▼
步骤3：买家确认收货
┌──────────────────────────────────────────────────────┐
│ 买家点击"确认收货"                                     │
│ → 订单状态：WAITING_MEET → COMPLETED                  │
│ → 卖家钱包余额 += 商品价格（资金解冻，打入卖家账户）      │
│ → 写入 wallet_log：INCOME，金额 = +商品价格             │
│ → 买家 EXP +50，卖家 EXP +50                          │
│ → 商品状态：PENDING → SOLD                            │
│                                                      │
│ 🔒 以上所有数据库操作必须在同一个 Transaction 中完成！    │
└──────────────────────────────────────────────────────┘
```

**订单取消（退款）流程**：

```
买家在确认收货前可取消订单
      │
      ▼
┌──────────────────────────────────────────────────────┐
│ → 订单状态：WAITING_MEET → CANCELLED                   │
│ → 买家钱包余额 += 商品价格（退款）                       │
│ → 写入 wallet_log：REFUND，金额 = +商品价格             │
│ → 商品状态：PENDING → ON_SALE（重新上架）               │
│                                                      │
│ 🔒 以上所有数据库操作必须在同一个 Transaction 中完成！    │
└──────────────────────────────────────────────────────┘
```

**订单状态流转图**：

```
                    ┌──────────────┐
                    │ WAITING_MEET │ 待线下见面
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │ 买家取消    │            │ 买家确认收货
              ▼            │            ▼
       ┌──────────┐        │     ┌───────────┐
       │ CANCELLED │        │     │ COMPLETED │
       │  已取消   │        │     │  已完成    │
       └──────────┘        │     └───────────┘
                            │
              (订单状态不会直接到这里)
```

**事务代码示例（技术要点）**：

```java
@Transactional(rollbackFor = Exception.class)
public void confirmReceipt(Long orderId, Long buyerId) {
    // 1. 查询订单，校验状态和买家身份
    TradeOrder order = orderMapper.selectById(orderId);
    if (!order.getStatus().equals("WAITING_MEET")) {
        throw new BusinessException("订单状态异常");
    }
    if (!order.getBuyerId().equals(buyerId)) {
        throw new BusinessException("无权操作");
    }

    // 2. 更新订单状态
    order.setStatus("COMPLETED");
    orderMapper.updateById(order);

    // 3. 卖家钱包加钱
    SysUser seller = userMapper.selectById(order.getSellerId());
    seller.setWalletBalance(
        seller.getWalletBalance().add(order.getPrice())
    );
    userMapper.updateById(seller);

    // 4. 记录卖家收入流水
    WalletLog incomeLog = new WalletLog();
    incomeLog.setUserId(order.getSellerId());
    incomeLog.setType("INCOME");
    incomeLog.setAmount(order.getPrice());
    incomeLog.setBalanceAfter(seller.getWalletBalance());
    incomeLog.setOrderId(orderId);
    walletLogMapper.insert(incomeLog);

    // 5. 买卖双方加经验值
    addExp(order.getBuyerId(), 50);
    addExp(order.getSellerId(), 50);

    // 6. 更新商品状态
    Item item = itemMapper.selectById(order.getItemId());
    item.setStatus("SOLD");
    itemMapper.updateById(item);

    // 任何一步出错 → 全部回滚（@Transactional 保证）
}
```

**验收标准**：
- [ ] 下单时买家余额正确扣减
- [ ] 确认收货时卖家余额正确增加
- [ ] 取消订单时买家收到退款
- [ ] 事务保证数据一致性（模拟异常场景验证回滚）
- [ ] 同一商品不能被重复购买

## 附录C：AI 审核流程详解

### C.1 Prompt 设计（参考模板）

```
你是一个校园二手交易平台的内容审核助手。请分析以下用户发布的商品信息，判断是否合规。

商品标题：{title}
商品描述：{description}
发布类型：{type}（SELL=出售，BUY=求购）

审核规则：
1. 严禁发布：危险品、管制物品、代考代写服务、虚假信息
2. 严禁发布：含人身攻击、侮辱性词汇、污言秽语的内容
3. 严禁发布：与校园交易无关的广告、引流信息
4. 明显价格欺诈（如市场价5000元的全新iPhone标价100元）也视为违规

请严格按照以下 JSON 格式返回（不要包含任何其他文字）：
{
  "is_violation": true或false,
  "violation_type": "违规类型（如违禁品/代考代写/人身攻击/虚假信息/无关内容，如合规则为null）",
  "violation_reason": "违规原因简述（如合规则为null）",
  "tags": ["标签1", "标签2", "标签3"]
}

标签提取规则（如合规）：
- 提取商品的关键属性词，如品牌、型号、成色、品类等
- 标签数量为 3-6 个
- 每个标签不超过 5 个字
- 例子：描述"99新苹果平板" → tags: ["iPad", "苹果", "99新", "平板"]
```

### C.2 后端处理伪代码

```java
public ItemPublishResult publishItem(ItemPublishDTO dto, Long userId) {
    // 1. 调用 AI 审核
    AiReviewResult aiResult;
    try {
        aiResult = aiService.review(dto.getTitle(), dto.getDescription());
    } catch (AiTimeoutException | AiParseException e) {
        // AI 异常 → 记录待人工复核，商品默认放行
        ViolationReport report = buildReport(dto, userId, "AI审核异常，待人工复核");
        report.setAiReviewError(true);
        violationReportMapper.insert(report);
        return publishItemDirectly(dto, userId, false);
    }

    // 2. 判定违规 → 拦截
    if (aiResult.isViolation()) {
        ViolationReport report = buildReport(dto, userId,
            aiResult.getViolationType(), aiResult.getViolationReason());
        violationReportMapper.insert(report);
        throw new BusinessException("发布内容涉嫌违规，已被拦截。如有疑问请联系客服。");
    }

    // 3. 合规 → 打标签并发布
    Item item = buildItem(dto, userId);
    item.setAiTags(JSON.toJsonString(aiResult.getTags()));
    item.setAiReviewed(true);
    item.setStatus("ON_SALE");
    itemMapper.insert(item);

    return new ItemPublishResult(true, "发布成功", item.getId());
}
```

### C.3 AI 接口异常处理策略

| 异常场景 | 处理策略 | 用户体验 |
|----------|----------|----------|
| 接口超时（> 5s） | 商品默认放行，`ai_reviewed = false`，标记待人工复核 | 正常发布，用户无感知 |
| 返回格式无法解析 | 同上 | 正常发布，用户无感知 |
| 接口返回 5xx 错误 | 同上 | 正常发布，用户无感知 |
| 接口返回 4xx 错误（鉴权失败等） | 记录日志，通知管理员排查 | 正常发布，用户无感知 |
| 明确违规 | 拦截发布，写入违规记录 | 提示违规，引导修改 |

> 📌 **核心原则**：AI 只做"减法"（拦截明显违规内容），不做"加法"（不放行可能导致更严重问题）。当 AI 无法正常工作时，宁可放行再由人工复核，也不能阻塞正常用户的发布流程。
